## Main idea

`Try 54` replaces the single large regressor with:

- `6` topology experts
- one expert per topology class
- all antenna heights inside each expert

The architecture used by every expert is `PMHHNet`.

## What is GT here

There are two different kinds of supervision in `Try 54`:

1. path-loss GT
   - comes directly from the dataset field `path_loss`
   - this is the true regression target of every expert

2. routing pseudo-GT
   - comes from the deterministic topology partition heuristic
   - it is derived from `topology_map` statistics, not predicted by a model

So the classifier is **not** a prerequisite for expert training.

Instead:

- experts are trained first with heuristic partitions
- the classifier is trained later to reproduce that same routing policy
- inference can then become classifier -> registry -> expert

## PMHHNet

`PMHHNet` is:

- a `PMNet` residual backbone
- plus a high-frequency branch
- plus FiLM-style continuous antenna-height conditioning

This lets us avoid splitting experts by height while still letting the network
change behavior continuously with antenna altitude.

## No-data handling

The current dataset does **not** contain an explicit HDF5 field for
`path_loss_no_data_mask`.

The pipeline is therefore set up in two tiers:

1. preferred source
   - `data.path_loss_no_data_mask_column`
   - if that field exists in the HDF5, it becomes the `no_data` target

2. current fallback
   - `data.derive_no_data_from_non_ground: true`
   - the target is derived from the topology-based `non_ground_mask`

This means:

- the path-loss regression still optimizes only on valid path-loss pixels
- but excluded pixels now still contribute through an auxiliary `no_data` head
- we do **not** need to pretend that all missing ray-tracing supervision equals
  exactly `180 dB`

## Expert outputs

Each expert now predicts two channels:

1. residual over the calibrated prior
2. `no_data` logit

The final path-loss prediction is still:

- `prior + residual`

The second head is only auxiliary and is trained with binary cross entropy.

## Validation and live-progress JSON

The validation JSON is now written in a cleaner grouped structure:

- `metrics.path_loss`
- `metrics.prior_path_loss`
- `metrics.improvement_vs_prior`
- `metrics.no_data`
- `focus`
- `experiment`
- `support`
- `runtime`
- `checkpoint`
- `selection`

Important additions:

- direct RMSE/MAE improvement versus prior
- `no_data` accuracy / precision / recall / IoU
- experiment summary with topology class, model size, sample count, and the
  resolved `no_data` target source used at runtime
- `selection` with metric name, current score, and whether the epoch is the
  best so far
- timing, throughput, and learning-rate information in `runtime`
- `focus.regimes` filtered to the routed city-type family of the active expert, plus
  global `LoS/NLoS` slices, so validation JSONs do not get flooded with
  irrelevant morphology buckets

In addition, the trainer now writes:

- `train_progress_latest.json`

while the epoch is still running.

That live file includes:

- current epoch and step
- progress fraction
- current `train_batches_per_second`
- current `seconds_per_train_batch`
- running generator loss
- running no-data loss
- current learning rate

## Fast training profile

The generated expert configs currently target fast iteration:

- `image_size = 513`
- `batch_size = 2`
- `base_channels = 40`
- `hf_channels = 16`
- `encoder_blocks = [1, 2, 2, 2]`
- `save_every = 1`

Checkpoint retention is intentionally minimal:

- keep `best_model.pt`
- keep the latest `epoch_N_model.pt`
- delete the previous epoch checkpoint automatically

## Cluster intent

The intended UPC launch pattern for `Try 54` is:

- upload `TFGFiftyFourthTry54`
- submit the `6` experts as independent `2 GPU` jobs
- keep outputs separated by topology class

This is a better match than the previous chained stages because:

- the experts are independent
- there is no teacher dependency between them
- and scheduler parallelism can help if free GPUs appear

## Current speed optimizations

The branch now includes a few low-risk speed optimizations aimed at the real
training bottleneck:

- formula-prior cache enabled in the expert YAMLs
- cluster jobs override the cache to a node-local temporary directory and
  precompute it before the first training epoch
- `prefetch_factor` enabled on the train loader as well as validation
- `cudnn.benchmark = true` on CUDA runs with fixed image size

This is intentionally conservative:

- no `torch.compile`
- no channels-last rewrite
- no architecture shrink during the same experiment

The priority was to accelerate the repeated physical-prior work without making
the training setup fragile.

Importantly, this cache is **not** another heavy HDF5 artifact. It is a
temporary per-job collection of `.pt` tensors on local scratch.

The expert launcher now also removes that temporary cache explicitly at job
exit so it does not accumulate under `SLURM_TMPDIR`.
